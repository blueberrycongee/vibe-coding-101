# 05 多人协作与规范

## 协作原则

### 1. 共识先行：目标与边界优先对齐

在团队中使用 vibecoding，最大的风险不是 AI 的能力问题，而是**每个人对 AI 的使用方式不一致**。

核心问题：
- 什么代码可以直接用 AI 生成？什么必须手写？
- AI 生成的代码需要经过什么级别的审查？
- 项目规则文件（CLAUDE.md / .cursorrules）谁来维护？

### 2. 责任清晰：AI 代码的责任归属

Boris Cherny 的立场很明确：AI 代码和人类代码使用同一质量标准，**提交者对代码负责**，不管是谁（人还是 AI）写的。

这意味着：
- 使用 AI 生成代码的开发者有责任审查和理解这些代码
- "AI 写的"不是降低审查标准的理由
- 代码出问题时，追责的是提交者，不是 AI

### 3. 资料可追溯：引用、链接与版本记录

- Prompt 模板和 slash commands 应纳入版本控制
- 重要决策的 AI 对话记录应保留（或摘要保留）
- 复盘结论应写入共享文档

## 团队 Vibecoding 实践

### 统一项目规则文件

团队应共同维护项目规则文件，并提交到 git：

```markdown
# CLAUDE.md（团队共享版）

## 代码规范
- [列出团队统一的编码规范]

## 架构约束
- [列出不可违反的架构决策]

## AI 使用规范
- 生成的代码必须通过 CI 检查
- 安全相关代码必须经过人工 review
- 不要让 AI 修改 CI/CD 配置
```

Boris 建议：每次 AI 犯错后更新 CLAUDE.md，把教训转化为规则，团队共享这些经验。（参见：[[invest-in-claude-md]]）

### 共享 Slash Commands

把团队的标准工作流做成 slash commands，提交到 git：
- `/commit` — 标准化提交流程
- `/pr` — PR 创建模板
- `/review` — 代码审查流程
- `/feature` — 功能开发标准流程

（参见：[[slash-commands-standardize]]）

### 并行开发模式

Boris 的并行工作流也适用于团队：
- 每人在独立的 git worktree 中工作
- 各自运行 AI 会话，处理独立的子任务
- 定期合并和同步

（参见：[[parallel-sessions-worktrees]]）

## 共享基线

### 最小质量标准
- [ ] 代码通过 lint 和类型检查
- [ ] 关键路径有测试覆盖
- [ ] 无已知安全漏洞
- [ ] 符合项目规则文件中的约束

### 团队术语表
在 CLAUDE.md 中维护团队术语表，确保 AI 和团队成员使用一致的命名：

```markdown
## 术语
- "用户"指注册用户，不包含匿名访客
- "订单"仅指已支付的订单
- "管理员"指有 admin 角色的用户
```
