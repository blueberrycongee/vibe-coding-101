# 03 提示词与任务拆解

## 提示词的核心结构

好的 prompt 不是”写得越详细越好”，而是要**传达足够的上下文让 AI 做出正确决策**。

### 基本结构：目标-约束-输出

```
目标：你要达成什么（一句话）
约束：时间、范围、风格、质量、技术栈
输出：格式、长度、示例
```

### 实际案例

**差的 prompt**：
> 帮我写一个登录页面

**好的 prompt**：
> 为现有的 Next.js 14 项目创建一个登录页面。要求：
> - 使用 app router 和 server actions
> - 表单包含邮箱和密码字段，使用 zod 做验证
> - 样式用已有的 Tailwind + shadcn/ui 组件
> - 错误信息要显示在表单下方
> - 不需要注册功能，只做登录

关键区别：好的 prompt 提供了**技术上下文**（Next.js 14, app router）、**范围边界**（不需要注册）和**具体约束**（zod, shadcn/ui）。

## 高级 Prompt 技巧

### 让 AI 先提问

Boris Cherny 在使用 plan mode 时，会明确要求 AI 先提出澄清问题：

> “在开始实现之前，请先列出你需要确认的问题。”

这能暴露你没想到的边界条件和假设。（参见：[[plan-mode-workflow]]）

### 用项目规则文件替代重复 prompt

与其每次在 prompt 中重复项目约束，不如把它们写入 CLAUDE.md（Claude Code）或 .cursorrules（Cursor）：

```markdown
# CLAUDE.md
- 本项目使用 TypeScript strict mode
- 测试框架使用 Vitest
- 样式使用 Tailwind CSS，不要用 inline style
- 组件放在 src/components/ 下，按功能分目录
```

这样每次对话 AI 都会自动遵守这些规则。（参见：[[invest-in-claude-md]]）

## 任务拆解

### 拆解原则

- 把需求分成 **3-5 个可验证的子任务**
- 每个子任务都要有明确的**可交付物**（一个文件、一个函数、一个测试通过）
- 子任务之间的**依赖关系**要清晰

### 拆解模板

```
任务：[一句话描述]

子任务：
1. [子任务名] → 交付物：[具体产物]
2. [子任务名] → 交付物：[具体产物]
3. [子任务名] → 交付物：[具体产物]

依赖：2 依赖 1 的接口定义
验收标准：[如何判断整体任务完成]
```

### 实际案例：实现用户认证

```
任务：为 API 添加 JWT 认证

子任务：
1. 定义认证中间件接口 → 交付物：middleware/auth.ts 类型定义
2. 实现 JWT 签发和验证逻辑 → 交付物：lib/jwt.ts + 单元测试通过
3. 实现登录端点 → 交付物：POST /api/login 端点 + 集成测试
4. 为受保护路由添加中间件 → 交付物：受保护端点返回 401（未认证）或 200（已认证）
5. 错误处理和边界情况 → 交付物：token 过期、格式错误等场景的测试

依赖：2→1, 3→2, 4→2
验收标准：所有测试通过 + 手动测试登录流程正常
```

## 常见反模式

| 反模式 | 问题 | 改进 |
|--------|------|------|
| 一个 prompt 做所有事 | AI 容易失去焦点，遗漏细节 | 拆分为多个子任务 |
| 没有给 AI 上下文 | AI 做出不符合项目风格的选择 | 使用项目规则文件 |
| 过度指定实现细节 | 限制了 AI 找到更好方案的空间 | 描述”做什么”而非”怎么做” |
| 忽略否定约束 | AI 可能添加不需要的功能 | 明确说”不需要 X” |
